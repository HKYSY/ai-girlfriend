## 一、产品形态

### 1.1 表现形式

**AI 女友**是一款基于 Live2D 虚拟形象 + 大语言模型（LLM）的**桌面端 Web 应用**，定位为"有情感记忆的虚拟伴侣"。

核心特征：
- **Live2D 虚拟形象驱动**：使用 Cubism 4 格式的 Live2D 模型，支持表情、动作、视线跟随、物理摆动，提供拟人化的视觉陪伴感。
- **情感化对话**：AI 不是简单的问答机器人，而是有 10 级心情系统、8 种情绪标签、性格模板的"有情绪的伴侣"，回复会随心情变化。
- **桌宠养成系统**：包含金币、饱腹感、疲劳度、亲密度、签到、商店、约会、小游戏等养成玩法，让互动有持续目标。
- **长期记忆与成长**：对话历史持久化、心情历史记录、成就系统、AI 自动写日记，形成"共同回忆"。

### 1.2 终端形态

| 维度 | 说明 |
|------|------|
| 运行终端 | **桌面浏览器**（Chrome/Edge 等 Chromium 内核浏览器优先）或 **Electron 桌面应用** |
| 部署模式 | **本地部署**（前后端均运行在用户本机，数据存储在本地） |
| 访问方式 | 开发期 `http://localhost:5173`（Vite 开发服务器），生产期 `http://localhost:3001`，桌面版双击 exe |
| 屏幕适配 | 桌面分辨率（≥1280×720），暂不适配移动端 |
| 联网需求 | 需联网调用 DeepSeek API；本地 Live2D 渲染与数据存储无需联网 |

### 1.3 核心价值主张

> "她记得你说过的话，会因为你开心而开心，也会因为你冷落她而撒娇生气——不只是一个聊天框，而是一个会成长、会记日记、会陪你玩游戏的虚拟伴侣。"

### 2.3 外部服务与 SDK

| 服务 | 用途 | 接入方式 |
|------|------|----------|
| **DeepSeek API** | 大语言模型对话生成（流式 + 非流式） | OpenAI 兼容格式，`https://api.deepseek.com/v1/chat/completions` |
| **Live2D Cubism Core** | Live2D 模型运行时 | 本地 SDK（`live2dcubismcore.min.js`），通过 `<script>` 引入 |

### 2.4 数据存储

**当前方案：SQLite 数据库（`backend/data/app.db`）**

| 数据表 | 说明 |
|--------|------|
| `characters` | 角色元数据（名字、性格模板、模型、心情等） |
| `messages` | **对话消息（永久保留，不自动删除）** |
| `conversation_meta` | 对话元信息（摘要、最后心情、最后活跃时间） |
| `memory_facts` | 事实记忆（AI 自动提取的关键信息：生日、约定、喜好等） |
| `memory_summaries` | 长期摘要（旧对话压缩存档） |
| `pet_state` | 桌宠状态（金币、饱腹感、疲劳度、亲密度、成就等） |
| `mood_history` | 心情历史（时间序列数据点，30 天保留） |
| `diary` | AI 日记（每天一篇，90 天保留） |

> 📁 旧 JSON 文件（`data/characters.json`、`data/conversations/` 等）在首次启动时自动迁移到 SQLite，原文件保留在 `data/` 目录下作为备份。

**对话备份**：
- 每次聊天后自动导出完整对话快照到 `data/backup/` 目录
- 每个角色最多保留 50 份备份
- 格式：JSON，便于用户直接查看和迁移

## 三、功能范围（核心业务定义）

### 3.1 核心功能模块

#### 模块 A：角色管理
- 多角色创建与切换
- 性格模板选择（玉子/自定义）
- Live2D 模型绑定（预设模型 + 用户上传模型）
- 角色删除（含关联数据级联清理）

#### 模块 B：Live2D 交互
- 模型渲染（Cubism 4 格式）
- 拖拽移动、滚轮缩放、双击重置位置
- 表情切换、动作播放（手动切换 + 互动触发）
- 视线跟随鼠标（可开关）
- 对话气泡（互动时弹出情绪文字）
- 分栏布局（Live2D 区与聊天区可拖拽调整比例）

#### 模块 C：情感化对话
- SSE 流式输出（逐字显示）
- 10 级心情系统（影响 AI 回复语气）
- 8 种情绪标签（开心/生气/难过/撒娇/惊讶/疑惑/害羞/平静）
- 心情自动衰减（长时间不聊心情下降）
- AI 主动发消息（5 分钟无互动触发）
- **对话历史永久保留**（数据库存储，不自动删除）
- **对话分级发送（Token 优化）**：最近 10 条原文 → 11-25 条截断 100 字 → 26 条以上仅看摘要，每次聊天节省约 60% token
- 对话摘要存档（超 50 条触发摘要生成，保留长期记忆）
- 情感分析兜底（AI 未输出心情标记时，自动分析用户消息情感）

#### 模块 D：桌宠养成系统
- **状态系统**：金币、饱腹感、疲劳度、亲密度
- **每日签到**：签到得金币
- **商店**：购买食物/饮料/礼物恢复状态
- **约会**：逛街/看电影等互动提升亲密度
- **小游戏**：猜拳、猜数字、幸运转盘（赌博逻辑）
- **状态衰减**：长时间不互动，饱腹感下降、疲劳度恢复
- **成就系统**：8 类成就多档位，自动检测解锁

#### 模块 E：记忆与成长
- **心情历史图**：折线图展示 1/7/30 天心情趋势
- **AI 日记**：每天首次打开页面自动生成**昨天**的日记。字数根据当天聊天量动态调整（50-400 字）。一天一篇，不追加。
- **事实记忆**：AI 自动从对话中提取关键事实（生日、约定、喜好、事件等），7 种事实类型
- **智能搜索**：支持按关键词搜索历史聊天记录
- **消息分页**：聊天记录分页加载，支持翻看全部历史
- **成就解锁记录**：里程碑追踪

#### 模块 F：模型管理
- 预设模型库（haru、icegirl）
- 用户上传模型（支持 7z/zip 压缩包，自动解压）
- 用户模型删除

### 3.2 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | Live2D 渲染 60fps；首条 AI 回复 < 3 秒首字 |
| 数据安全 | API Key 仅存于后端 `.env`，不暴露给前端 |
| 用户体验 | 流式输出避免等待焦虑；状态可视化直观 |
| 可维护性 | TypeScript 强类型；模块化组件拆分 |
| 存储可靠性 | SQLite WAL 模式 + 对话 JSON 备份，双重保障 |

## 四、部署要求

### 4.1 开发环境搭建

#### 前置条件
- **Node.js** ≥ 18（推荐 20 LTS）
- **npm**
- **DeepSeek API Key**（从 DeepSeek 开放平台获取）

#### 步骤

```bash
# 1. 后端配置
cd backend
# 复制 .env.example 为 .env，填入 DEEPSEEK_API_KEY
npm install
npm run build         # 编译 TypeScript
npm start             # 启动后端（端口 3001）

# 2. 前端配置（新开终端）
cd frontend
npm install
npm run dev           # 启动前端（Vite，端口 5173）
```

#### 访问
- 开发：浏览器打开 `http://localhost:5173`
- Vite 自动将 `/api/*` 请求代理到 `http://localhost:3001`

### 4.2 环境变量说明

| 变量名 | 位置 | 必填 | 说明 |
|--------|------|------|------|
| `DEEPSEEK_API_KEY` | backend/.env | 是 | DeepSeek API 密钥 |
| `DEEPSEEK_MODEL` | backend/.env | 否 | 模型名，默认 `deepseek-chat` |
| `PORT` | backend/.env | 否 | 后端端口，默认 3001 |

### 4.3 生产部署

```bash
# 前端构建
cd frontend
npm run build        # 产物在 frontend/dist/

# 后端构建
cd backend
npm run build        # 产物在 backend/dist/

# 启动
cd backend
npm start            # node dist/index.js
```

> **扩展建议**：生产环境可让后端 Express 托管 `frontend/dist` 静态资源，实现单端口访问（当前未配置，列为后期规划）。

## 六、开发规范

### 6.1 技术决策原则

1. **本地优先**：默认所有数据存储在本地，保障用户隐私和数据可控。
2. **类型安全**：前后端均使用 TypeScript，禁止 `any`（除第三方库无类型声明外，需加 `@ts-expect-error` 注释）。
3. **ESM 模块**：前后端均使用 ES Modules，后端导入本地文件须带 `.js` 后缀。
4. **声明式扩展**：成就、商店、约会等配置采用声明式数组定义，新增内容不改逻辑只改配置。
5. **SSE 流式优先**：所有 AI 回复采用 Server-Sent Events 流式输出，保证用户体验。
6. **数据库级联**：删除角色时通过 `ON DELETE CASCADE` 自动清理关联数据。

### 6.2 架构约定

| 约定 | 说明 |
|------|------|
| z-index 层级 | `chat-panel(2) < canvas(3) < overlay(4) < sidebar(5) < interact-panel(9999)` |
| API 路由前缀 | 所有后端接口以 `/api/` 开头 |
| 数据隔离 | 按角色 ID 隔离数据，通过数据库表 `characterId` 外键关联 |
| 心情范围 | 0-100，单次变化不超过 ±10（`clampMoodChange`） |
| 对话保留 | 数据库永久保留，内存上下文保持最近 80 条（`MAX_HISTORY_ROUNDS * 2`） |

### 6.3 变更管理

- **任何技术方案变更**（新增依赖、调整架构、改变数据格式）须先更新本文档。
