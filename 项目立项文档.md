## 一、产品形态

### 1.1 表现形式

**AI 女友**是一款基于 Live2D 虚拟形象 + 大语言模型（LLM）的**桌面端 Web 应用**，定位为"有情感记忆的虚拟伴侣"。

核心特征：
- **Live2D 虚拟形象驱动**：使用 Cubism 4 格式的 Live2D 模型，支持表情、动作、视线跟随、物理摆动，提供拟人化的视觉陪伴感。
- **情感化对话**：AI 不是简单的问答机器人，而是有 10 级心情系统、8 种情绪标签、性格模板的"有情绪的伴侣"，回复会随心情变化。
- **桌宠养成系统**：包含金币、饱腹感、疲劳度、亲密度、签到、商店、约会、小游戏等养成玩法，让互动有持续目标。
- **长期记忆与成长**：对话历史持久化、心情历史记录、成就系统、AI 自动写日记，形成"共同回忆"。

### 1.2 终端形态

| 维度 | 说明 |
|------|------|
| 运行终端 | **桌面浏览器**（Chrome/Edge 等 Chromium 内核浏览器优先） |
| 部署模式 | **本地部署**（前后端均运行在用户本机，数据存储在本地文件系统） |
| 访问方式 | 开发期 `http://localhost:5173`（Vite 开发服务器），生产期可由后端统一托管前端静态资源 |
| 屏幕适配 | 桌面分辨率（≥1280×720），暂不适配移动端 |
| 联网需求 | 需联网调用 DeepSeek API；本地 Live2D 渲染与数据存储无需联网 |

### 1.3 核心价值主张

> "她记得你说过的话，会因为你开心而开心，也会因为你冷落她而撒娇生气——不只是一个聊天框，而是一个会成长、会记日记、会陪你玩游戏的虚拟伴侣。"

| 服务 | 用途 | 接入方式 |
|------|------|----------|
| **DeepSeek API** | 大语言模型对话生成（流式 + 非流式） | OpenAI 兼容格式，`https://api.deepseek.com/v1/chat/completions` |
| **Live2D Cubism Core** | Live2D 模型运行时 | 本地 SDK（`live2dcubismcore.min.js`），通过 `<script>` 引入 |

### 2.4 数据存储

**当前方案：JSON 文件持久化（无数据库）**

| 数据类型 | 存储位置 | 说明 |
|----------|----------|------|
| 角色列表 | `backend/data/characters.json` | 所有角色的元数据 |
| 对话历史 | `backend/data/conversations/{id}.json` | 每个角色一个文件 |
| 桌宠状态 | `backend/data/petstate/{id}.json` | 金币/饱腹感/疲劳/成就等 |
| 心情历史 | `backend/data/moodhistory/{id}.json` | 时间序列数据点，30天保留 |
| AI 日记 | `backend/data/diary/{id}.json` | 每日日记，90天保留 |
| 上传模型 | `backend/uploads/live2d/` | 用户上传的模型文件 |

> 选择文件存储而非数据库的原因：本地单机部署、数据量小、无并发写入需求、便于用户直接查看和迁移数据。

## 三、功能范围（核心业务定义）

### 3.1 核心功能模块

#### 模块 A：角色管理
- 多角色创建与切换
- 性格模板选择（温柔/活泼/傲娇/知性/自定义）
- Live2D 模型绑定（预设模型 + 用户上传模型）
- 角色删除（含关联数据清理）

#### 模块 B：Live2D 交互
- 模型渲染（Cubism 4 格式）
- 拖拽移动、滚轮缩放、双击重置位置
- 表情切换、动作播放（手动切换 + 互动触发）
- 视线跟随鼠标（可开关）
- 对话气泡（互动时弹出情绪文字）
- 分栏布局（Live2D 区与聊天区可拖拽调整比例）

#### 模块 C：情感化对话
- SSE 流式输出（逐字显示）
- 10 级心情系统（影响 AI 回复语气）
- 8 种情绪标签（开心/生气/难过/撒娇/惊讶/疑惑/害羞/平静）
- 心情自动衰减（长时间不聊心情下降）
- AI 主动发消息（5 分钟无互动触发）
- 对话历史持久化（保留最近 20 轮上下文）

#### 模块 D：桌宠养成系统
- **状态系统**：金币、饱腹感、疲劳度、亲密度
- **每日签到**：签到得金币
- **商店**：购买食物/饮料/礼物恢复状态
- **约会**：逛街/看电影等互动提升亲密度
- **小游戏**：猜拳、猜数字、幸运转盘（赌博逻辑）
- **状态衰减**：长时间不互动，饱腹感下降、疲劳度恢复
- **成就系统**：8 类成就多档位，自动检测解锁

#### 模块 E：记忆与成长
- **心情历史图**：折线图展示 1/7/30 天心情趋势
- **AI 日记**：每天首次打开自动生成第一人称日记
- **成就解锁记录**：里程碑追踪

#### 模块 F：模型管理
- 预设模型库（haru、icegirl）
- 用户上传模型（支持 7z/zip 压缩包，自动解压）
- 用户模型删除

### 3.2 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | Live2D 渲染 60fps；首条 AI 回复 < 3 秒首字 |
| 数据安全 | API Key 仅存于后端 `.env`，不暴露给前端 |
| 用户体验 | 流式输出避免等待焦虑；状态可视化直观 |
| 可维护性 | TypeScript 强类型；模块化组件拆分 |

## 四、部署要求

### 4.1 开发环境搭建

#### 前置条件
- **Node.js** ≥ 18（推荐 20 LTS）
- **npm** 或 **pnpm**
- **DeepSeek API Key**（从 DeepSeek 开放平台获取）

#### 步骤

```bash
# 1. 后端配置
cd backend
cp .env.example .env
# 编辑 .env，填入 DEEPSEEK_API_KEY
npm install
npm run dev          # 启动后端（tsx watch 热重载，端口 3001）

# 2. 前端配置（新开终端）
cd frontend
npm install
npm run dev          # 启动前端（Vite，端口 5173）
```

#### 访问
- 开发：浏览器打开 `http://localhost:5173`
- Vite 自动将 `/api/*` 请求代理到 `http://localhost:3001`

### 4.2 环境变量说明

| 变量名 | 位置 | 必填 | 说明 |
|--------|------|------|------|
| `DEEPSEEK_API_KEY` | backend/.env | 是 | DeepSeek API 密钥 |
| `DEEPSEEK_MODEL` | backend/.env | 否 | 模型名，默认 `deepseek-v4-flash` |
| `PORT` | backend/.env | 否 | 后端端口，默认 3001 |

### 4.3 生产部署

```bash
# 前端构建
cd frontend
npm run build        # 产物在 frontend/dist/

# 后端构建
cd backend
npm run build        # 产物在 backend/dist/

# 启动
cd backend
npm start            # node dist/index.js
```

> **扩展建议**：生产环境可让后端 Express 托管 `frontend/dist` 静态资源，实现单端口访问（当前未配置，列为后期规划）。

## 六、开发规范

### 6.1 技术决策原则

1. **本地优先**：默认所有数据存储在本地，保障用户隐私和数据可控。
2. **类型安全**：前后端均使用 TypeScript，禁止 `any`（除第三方库无类型声明外，需加 `@ts-expect-error` 注释）。
3. **ESM 模块**：前后端均使用 ES Modules，后端导入本地文件须带 `.js` 后缀。
4. **声明式扩展**：成就、商店、约会等配置采用声明式数组定义，新增内容不改逻辑只改配置。
5. **SSE 流式优先**：所有 AI 回复采用 Server-Sent Events 流式输出，保证用户体验。

### 6.2 架构约定

| 约定 | 说明 |
|------|------|
| z-index 层级 | `chat-panel(2) < canvas(3) < overlay(4) < sidebar(5) < interact-panel(9999)` |
| API 路由前缀 | 所有后端接口以 `/api/` 开头 |
| 数据隔离 | 按角色 ID 隔离数据（对话、状态、日记、心情历史各一个文件） |
| 心情范围 | 0-100，单次变化不超过 ±5（`clampMoodChange`） |
| 对话历史 | 保留最近 20 轮（`MAX_HISTORY_ROUNDS`） |

### 6.3 变更管理

- **任何技术方案变更**（新增依赖、调整架构、改变数据格式）须先更新本文档。
- **新增功能**须在「功能范围」章节登记。
